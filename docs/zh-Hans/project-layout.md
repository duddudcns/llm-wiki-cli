# 整理你的 wiki,以及使用已有的 wiki

[English](../en/project-layout.md) · [한국어](../ko/project-layout.md) · [日本語](../ja/project-layout.md) · **简体中文** · [Español](../es/project-layout.md) · [Français](../fr/project-layout.md)

## 两种整理文件夹的方式

默认情况下,`llmw init` 会把 `raw/`、`wiki/` 和后台用的文件夹直接放在你项目的主文件夹里。如果你想让项目文件夹保持整洁,也可以把这一切都收进一个叫 `ai-wiki/` 的子文件夹里：

```bash
llmw init --layout ai-wiki
```

```text
ai-wiki/
  raw/ wiki/ .llmw/            # 和之前一样的文件夹,只是往下嵌套了一层
.claude/skills/llm-wiki/       # 不管哪种方式,这个都还是放在你的主项目文件夹里
.claude-plugin/plugin.json     # 不管哪种方式,这个都还是放在你的主项目文件夹里
```

每个命令都会自动识别出你用的是哪种方式——第一次 `init` 之后,你就再也不用告诉它了。如果你已经用普通方式建好了笔记,那什么都不用改。

如果你是在项目文件夹之外运行 `llmw`(比如从一个脚本里运行),你可以直接告诉它去哪里找,它会自己识别出你用的是哪种文件夹结构：

```bash
llmw --root /path/to/project status
LLMW_ROOT=/path/to/project llmw status
```

## 配合你已经手动建好的 wiki 使用 `llmw`

也许你早在听说这个工具之前就已经有了一套自己的笔记,现在只是想让 `llmw` 开始接手管理它。这种情况下用 `--adopt`,而不是普通的 `init`：

```bash
llmw init --adopt                  # 或者：llmw init --layout ai-wiki --adopt
```

这会搭建好后台用的搜索索引,但它**绝对不会**创建或覆盖你任何已有的笔记或文件夹——就算你之后又加上 `--force` 重新跑一次也不会。你的设置文件也是一样受保护的,所以你做过的任何自定义设置都不会被意外重置。(普通的 `llmw init`,不带 `--adopt` 的话,表现是不一样的：它会创建一些起始笔记和文件夹,而 `--force` 会覆盖掉它们——所以普通版本只适合在一个全新的空项目里使用。)

## 让 `llmw` 适应用不同规则的笔记

如果你已有的笔记组织方式跟这个工具不太一样——比如有些重要文件放在 `wiki/` 文件夹之外,或者用的标签跟 `llmw` 预期的不一样——你完全不需要重新整理任何东西。只要按照上面说的接手 wiki 之后,调整一下一个小小的设置文件就行：

```toml
[paths]
# 除了正常的 wiki/ 文件夹之外,搜索时还应该包含的单独笔记文件——
# 比如放在最外层的一个索引文件或更新日志文件。
extra_root_pages = ["index.md", "log.md", "schema.md"]

[lint]
# 每篇笔记开头应该具备哪些信息。
# 内置的默认值是 ["type", "status", "created", "updated"];
# "updated" 这一项也接受写成 "last_updated" 的笔记。
required_frontmatter = ["type", "status", "last_updated"]
```

你已有的笔记完全不需要为此做任何改动。

## 配合 Obsidian 一起使用

每篇笔记都是纯文本文件,所以你也可以直接用大家常用的笔记软件 [Obsidian](https://obsidian.md/) 打开 `wiki/` 文件夹——你会得到它的可视化地图视图、"谁链接到了这里"视图,以及它的搜索功能,而且用的都是同一批笔记,完全不用放弃 AI 使用这些笔记的方式。

笔记之间链接的一些细节,特意设计得和 Obsidian 本身一致,包括一些比较棘手的边界情况：

- Obsidian 所有的链接写法都能识别：普通链接到另一篇笔记、带自定义显示名的链接、链接到笔记里的某个具体标题、以及把另一篇笔记内容"嵌入"进来的写法。
- 带文件夹路径的链接,理解方式和 Obsidian 一样——是相对于 wiki 的最外层,而不是相对于写这个链接的那篇笔记。
- 笔记开头也可以用几种不同格式列出"相关笔记",这几种格式都能正确识别。
- 文件名里带空格或特殊字符的链接(这在笔记是从别的工具导出来的时候很常见)也能正确匹配上。
- 指向 `wiki/` 文件夹之外的链接,会去实际的磁盘上核实一下,所以只有在它真的不存在的时候才会被标记为失效。

**和真正的 Obsidian 相比,有几处小小的不同：** 这个工具能识别几种额外的笔记间联系(比如上面提到的"相关笔记"列表),这是 Obsidian 自己的地图视图不会显示的,因为它们是这个工具特有的。另外,如果两篇笔记刚好在不同文件夹里用了完全相同的文件名,那么当一个链接没有指明具体文件夹时,这个工具和 Obsidian 有时都会挑错笔记。除此之外,两边的表现是一致的。
