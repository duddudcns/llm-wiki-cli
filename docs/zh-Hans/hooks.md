# 插件自带的安全保障机制

[English](../en/hooks.md) · [한국어](../ko/hooks.md) · [日本語](../ja/hooks.md) · **简体中文** · [Español](../es/hooks.md) · [Français](../fr/hooks.md)

安装 Claude Code 插件(见 [installation.md](installation.md))会自动开启四个安全保障功能。这几个都不是使用 `llmw` 的必需条件——它们只是锦上添花,这个工具本身不管有没有它们都能自己保护好自己。

## 功能一：防止 AI 用错误的方式改笔记

理论上没有什么能阻止 AI 助手无视这个工具、直接去改一篇 wiki 笔记,就跟改其他任何文件一样。一旦发生这种情况,你就会失去自动备份、失去"我为什么要改这个"的必填理由,也失去笔记格式是否正确的检查——而实际情况中,只要有别的指令在主导、又完全没提到这个工具,这种情况就真的会发生。

装了 Claude Code 插件之后,有一个安全检查会拦住这种情况：如果 AI 想用它平时那套文件编辑工具直接改一篇 wiki 笔记,这次编辑就会被挡下来(或者,如果你更喜欢的话,只是先弹出确认),同时会明确告诉它该用哪个 `llmw` 命令才对——这样它可以立刻用正确的方式重新试一次,而不会卡住。

这个检查只会关注针对使用了这个工具的项目里 wiki 笔记(以及笔记所依据的、未经改动的原始素材所在的 `raw/` 文件夹)的编辑——其他所有内容都完全不受影响,包括单纯的读取文件。

你可以按项目在 `.llmw/config.toml` 里关掉这个检查,或者调整它的严格程度——同一个设置项同时管着 wiki 笔记和 `raw/` 文件夹：

```toml
[hooks]
wiki_guard = "deny"  # 默认：拦下这次编辑,并说明正确的做法
# wiki_guard = "ask"   # 改成先弹出确认,而不是直接拦下
# wiki_guard = "off"   # 给这个项目关掉这项检查
```

在 Windows 上,这项检查需要装了"Git Bash"才能运行。如果没装,这项检查就是不会跑而已——`llmw` 自带的安全规则(每次改动都要有理由、编辑前先备份等等)不受影响,照样生效。

## 功能二：提醒 AI 在每次请求前先看一眼 wiki

一个记满了过去决定和教训的 wiki,只有在 AI 真的会在开始新工作之前去看一眼的时候才有用——而如果全凭它自己判断,它不见得每次都会想起来看。(这和你自己在笔记软件里手写 wiki 会遇到的问题是一样的：你认认真真把东西记了下来,可 AI 还是会重蹈你早就写下来的覆辙,因为没有什么东西提醒它去看一眼。)

另外,每次会话开始时还会有一个小提醒："这个项目的 wiki 里已经有 N 篇笔记了"(如果 wiki 已经存在的话),或者一句"你应该跑一下 `llmw init`"的提示(如果 wiki 还不存在)——这样即使是在一个全新的项目里,AI 从第一条消息开始就知道有这个工具可以用。

为了进一步解决这个问题,你发送的大多数消息也都会附带一句简短的提醒："这个项目有个 wiki——先去搜一搜。"这么做是刻意从简的：它不会靠关键词匹配去猜你这条消息是不是真的和 wiki 里的某些内容相关,因为这种自动猜测很容易漏掉那些只是措辞不同、但说的其实是同一件事的笔记。它只是每次都提醒 AI 去看一眼,真正的判断(以及真正的搜索)则交给 AI 自己来做。(像"ok"或"谢谢"这样非常简短的消息,会跳过这条提醒——因为那里并没有什么真正开始的工作,可以拿来对着 wiki 核对。)

光靠这个提醒本身只是个轻推——它不会拦下或拖慢你的请求,也没办法阻止 AI 继续往下做。而且,每次看到一模一样的这句话,也很容易被当成背景噪音直接无视掉。

所以下面还有第二层、更强一点的机制：一次会话里,AI 第一次想要编辑一个真实的项目文件(不是 wiki 笔记本身)、但还没搜索过的时候,这次编辑会被暂停一次,要求它要么先搜索,要么明确判断这个任务不需要搜索。这一整个会话最多只会触发一次——只要跑过一次搜索(哪怕只是顺手搜一下),或者过了这一次确认,后面的编辑就会照常进行。这仍然不是硬性锁死：AI 完全可以确认一下就继续,而实际上并没有真的去搜索。它带来的好处是,把一个容易被划过去的提醒,变成一个真正要做判断的时刻。

```toml
[hooks]
search_gate = "ask"  # 默认：暂停会话里第一次真正的编辑,直到搜索过或确认过为止
# search_gate = "off"  # 给这个项目关掉这项检查
```

## 功能三：提醒 AI 在做完之后把 wiki 更新上

一个 wiki 只有真的跟得上实际发生的事情才有用——就像 AI 有可能在开始工作前忘了看一眼 wiki 一样,它也完全可能认认真真做完了活儿,却什么都没写下来。

为了抓住这种情况,`llmw` 会按会话跟踪项目文件是不是在 wiki 上次被更新(通过 `llmw write`/`edit`/`patch`/`archive`)之后又发生了变化。如果 AI 想在这种状态下结束这一轮,就会被暂停一次,要求它要么记下改了什么、为什么改,要么明确判断这次改动不值得更新 wiki。跟上面的搜索检查一样,这个也是每一轮最多触发一次——Claude Code 自身的循环保护机制保证它绝不会卡成死循环重试——所以如果 wiki 还是没跟上,会在*下一轮*结束时再提醒一次,而不是每条消息都念叨。

```toml
[hooks]
update_gate = "ask"  # 默认：源码变了但 wiki 没变时,每一轮暂停一次
# update_gate = "off"  # 给这个项目关掉这项检查
```

## 功能四：让命令行工具本身保持最新

这个插件里包含一个小小的辅助程序,但真正干活的其实是另外单独装在你电脑上的一份 `llmw`。通过应用市场更新插件,**并不会**自动更新那份单独安装的 `llmw`——如果放着不管,你可能在不知不觉中一直用着一个过时的版本。

为了避免这种情况,每次会话开始时都会跑一个快速检查：它会拿你电脑上装的 `llmw` 版本,和插件所期望的版本对比一下。如果两者对不上——包括 `llmw` 压根还没装的情况——它就会自动帮你重新装上正确的版本。所以更新插件的同时也会让命令行工具保持同步,你什么都不用额外做。

当版本本来就一致的时候,这项检查只是一次很快的本地检查,不需要联网——真正的重新安装只会在版本不一致的少数情况下才会发生。
