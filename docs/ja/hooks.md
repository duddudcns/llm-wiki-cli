# プラグインに組み込まれた安全装置

[English](../en/hooks.md) · [한국어](../ko/hooks.md) · **日本語** · [简体中文](../zh-Hans/hooks.md) · [Español](../es/hooks.md) · [Français](../fr/hooks.md)

Claude Codeのプラグイン([installation.md](installation.md) 参照)をインストールすると、4つの自動安全機能がオンになります。どれも `llmw` を使うために必須というわけではありません。もともとそれ自体で自分の身を守れるツールに、さらに便利さを上乗せしているだけです。

## 機能1: AIが間違ったやり方でメモを編集するのを防ぐ

技術的には、AIアシスタントがこのツールを無視して、他の普通のファイルと同じようにwikiのメモを直接編集してしまうことを止めるものは何もありません。もしそうなってしまうと、自動バックアップも、「なぜこの変更をしたのか」を記録する必須のメモも、メモが正しく書けているかのチェックも、すべて失われてしまいます。そして実際のところ、何か他の指示が優先されていて、このツールのことがまったく触れられていない場面では、これは実際に起こりがちです。

Claude Codeのプラグインとしてインストールしていると、この状況を安全チェックが捕まえてくれます。AIが普段のファイル編集ツールを使ってwikiのメモを直接編集しようとすると、その編集はブロックされます(設定によっては、先に確認を求めるだけにもできます)。そして、代わりにどの `llmw` コマンドを使えばよいかがはっきり示されるので、AIは行き詰まることなく、すぐに正しいやり方でもう一度試せます。

このチェックが見ているのは、このツールを使っているプロジェクトのwikiのメモを狙った編集(それと、メモの元になっている手つかずの元資料が入った `raw/` フォルダへの編集)だけです。それ以外のもの(ファイルをただ読むことも含めて)には一切手を出しません。

このチェックはプロジェクトごとに、`.llmw/config.toml` でオフにしたり、厳しさを変えたりできます。同じ設定がwikiのメモと `raw/` フォルダの両方に効きます。

```toml
[hooks]
wiki_guard = "deny"  # デフォルト: 編集をブロックし、正しいやり方を説明する
# wiki_guard = "ask"   # ブロックする代わりに確認を求める
# wiki_guard = "off"   # このプロジェクトではこのチェックをオフにする
```

Windowsでは、このチェックが動くには「Git Bash」がインストールされている必要があります。入っていない場合は、このチェックだけが動かなくなります。それでも `llmw` 自体に組み込まれた安全ルール(変更には必ず理由が必要、編集前に必ずバックアップを取る、など)は変わらず働き続けます。

## 機能2: 毎回のリクエストの前に、AIにwikiを確認するよう促す

過去の決定や失敗をまとめたwikiも、AIが新しい作業を始める前にちゃんと目を通してくれなければ意味がありません。AI任せにしていると、いつもそれを思い出してくれるとは限りません。(これは、メモアプリで手作りのwikiを使っている人がよく経験する問題と同じです。せっかく几帳面に書き残しても、見るよう促すものが何もなければ、AIはすでに書いてある失敗を平気で繰り返してしまいます。)

さらに、セッションの最初には毎回、ちょっとしたお知らせも表示されます。すでにwikiがある場合は「このプロジェクトにはN個のメモがあるwikiがあります」、まだない場合は「`llmw init` を実行するとよいです」という一行のヒントです。これにより、まったく新しいプロジェクトであっても、AIは最初のメッセージからこのツールの存在を知ることができます。

これをさらに後押しするため、あなたが送るほとんどのメッセージには、「このプロジェクトにはwikiがある — まず探してみて」という短いリマインダーが一言添えられます。これはあえてシンプルにしてあります。あなたのメッセージがwikiの中の何かと本当に関係あるかどうかを、キーワードを照らし合わせて推測しようとはしません。そういう自動判定は、書き方が少し違うだけのメモを簡単に見逃してしまうからです。そうではなく、とにかくAIに確認しに行かせて、実際に関係あるかどうかの判断(そして実際に探す作業)は、AI自身に任せます。(「ok」や「ありがとう」のような、とても短いメッセージのときはこのリマインダーは表示されません。そこから始まる本当の作業がなく、wikiと照らし合わせて確認する必要がないからです。)

これだけだと軽い後押しに過ぎません。あなたのリクエストをブロックしたり遅らせたりすることは一切なく、AIの作業を止めることもできません。それに、毎回まったく同じ一文が表示されると、そのうち読み飛ばされやすくもなります。

そこで、その下にもう一段階、もう少し強い仕組みが用意されています。セッション中にAIが(wikiのメモ自体ではなく)実際のプロジェクトファイルを初めて編集しようとしたとき、まだ一度も検索していなければ、その編集を一度だけ止めて、先に検索するか、この作業には検索が不要だとはっきり判断するよう求められます。これはセッションにつき最大1回しか発生しません — 検索が(一度きりのものでも)実行された瞬間、またはこの確認を一度通過した後は、通常どおり編集できるようになります。それでも完全なロックではありません。AIは確認だけして、実際には検索せずに進むこともできます。この仕組みが与えてくれるのは、読み飛ばしやすいリマインダーの代わりに、実際に判断を下す瞬間を強制することです。

```toml
[hooks]
search_gate = "ask"  # デフォルト: セッション最初の実際の編集を、検索または確認するまで一度止める
# search_gate = "off"  # このプロジェクトではこのチェックをオフにする
```

## 機能3: 作業が終わったらwikiを更新するようAIに促す

wikiは、実際に起きたことにちゃんと追いついていて初めて役に立ちます。AIが作業を始める前にwikiを確認し忘れることがあるのと同じように、たとえ最初は良いつもりでいても、実際の作業を終えたあとにそれを何も書き残さないまま終わってしまうこともあります。

これを捕まえるため、`llmw` はセッションごとに、wikiが最後に更新されて(`llmw write`/`edit`/`patch`/`archive` によって)から、プロジェクトのファイルが変更されたかどうかを追跡しています。もしAIがその状態のままターンを終えようとすると、一度だけ止められて、何をなぜ変更したかを記録するか、あるいはこの変更はwikiの更新が不要だとはっきり判断するよう求められます。上の検索チェックと同じく、これもターンにつき最大1回しか発生しません — Claude Code自体のループ防止機能のおかげで、抜け出せない再試行ループになることは決してありません — なので、wikiがまだ追いついていなければ*次の*ターンの終わりに再び促されます。メッセージのたびに毎回しつこく催促するわけではありません。

```toml
[hooks]
update_gate = "ask"  # デフォルト: ソースは変わったのにwikiが変わっていないとき、ターンにつき一度止める
# update_gate = "off"  # このプロジェクトではこのチェックをオフにする
```

## 機能4: コマンドラインツール自体を自動的に最新の状態に保つ

このプラグインには小さな補助プログラムが含まれていますが、実際の作業をしているのはパソコンにインストールされている、別のコピーの `llmw` です。マーケットプレイスからプラグインを更新しても、その別のコピーは自動的には更新され**ません**。放っておくと、気づかないうちに古いバージョンを使い続けてしまうことになりかねません。

これを防ぐため、セッションの最初に毎回ちょっとしたチェックが走ります。パソコンにインストールされている `llmw` のバージョンと、プラグインが期待しているバージョンを見比べるのです。もし食い違っていたら(そもそも `llmw` がまだインストールされていない場合も含めて)、正しいバージョンを自動的に入れ直してくれます。つまり、プラグインを更新するだけで、コマンドラインツールの方も自動的につじつまが合うようになっていて、あなたが何かする必要はありません。

バージョンがすでに一致している場合、このチェックはインターネット接続も不要な、素早いその場限りの確認で終わります。実際に入れ直す処理が走るのは、何かが食い違っているまれなときだけです。
