# 위키 정리하기, 그리고 이미 있는 위키 쓰기

## 폴더를 정리하는 두 가지 방법

기본적으로 `llmw init`은 `raw/`, `wiki/`, 그리고 뒤에서 쓰이는 폴더를
프로젝트의 메인 폴더에 바로 만듭니다. 프로젝트 폴더를 깔끔하게 유지하고
싶다면, 이 모든 걸 `ai-wiki/`라는 하위 폴더 하나에 몰아넣을 수도
있습니다:

```bash
llmw init --layout ai-wiki
```

```text
ai-wiki/
  raw/ wiki/ .llmw/            # 내용은 그대로, 한 단계 아래로 옮겨졌을 뿐
.claude/skills/llm-wiki/       # 이건 어느 방식이든 메인 프로젝트 폴더에 그대로 들어감
.claude/rules/llm-wiki.md      # 이것도 어느 방식이든 메인 프로젝트 폴더에 그대로 들어감
.claude-plugin/plugin.json     # 이것도 어느 방식이든 메인 프로젝트 폴더에 그대로 들어감
```

어떤 명령어를 쓰든 지금 어떤 방식을 쓰고 있는지 자동으로 알아냅니다.
처음 `init`을 한 번 실행한 뒤에는 다시 알려줄 필요가 없습니다. 이미
기본 방식으로 메모를 만들어 뒀다면 아무것도 바꿀 필요가 없습니다.

프로젝트 폴더 바깥에서 `llmw`를 실행한다면(예를 들어 스크립트에서),
그냥 어디를 봐야 하는지만 알려주면 두 방식 중 어느 쪽을 쓰고 있는지
알아서 파악합니다:

```bash
llmw --root /path/to/project status
LLMW_ROOT=/path/to/project llmw status
```

## 이미 직접 만들어 둔 위키와 함께 쓰기

이 도구를 알기 전부터 이미 자기만의 메모를 정리해 왔고, `llmw`가 그걸
그대로 관리해주기만 하면 되는 상황일 수도 있습니다. 그럴 때는 그냥
`init` 대신 `--adopt`를 쓰세요:

```bash
llmw init --adopt                  # 또는: llmw init --layout ai-wiki --adopt
```

이 명령어는 뒤에서 쓰이는 검색 인덱스는 만들어 주지만, 기존에 있던
메모나 폴더는 절대 새로 만들거나 덮어쓰지 않습니다. 나중에 `--force`를
붙여서 다시 실행해도 마찬가지입니다. 설정 파일도 똑같이 보호되기
때문에, 직접 만들어둔 설정이 실수로 초기화되는 일도 없습니다. (반면
`--adopt` 없이 그냥 `llmw init`을 쓰면 다르게 동작합니다. 이 경우엔
몇 가지 기본 메모와 폴더를 새로 만들고, `--force`를 붙이면 그것들을
덮어씁니다. 그러니 이 기본 방식은 완전히 비어 있는 새 프로젝트에서만
쓰세요.)

## 규칙이 조금 다른 메모에 맞추기

이미 있는 메모들이 조금 다른 방식으로 정리돼 있을 수도 있습니다.
예를 들어 중요한 파일 몇 개가 `wiki/` 폴더 바깥에 있거나, `llmw`가
기대하는 것과 다른 이름표를 쓰고 있을 수도 있죠. 그렇다고 다시
정리할 필요는 없습니다. 위에서 설명한 대로 위키를 먼저 가져온 다음,
작은 설정 파일만 살짝 손봐주면 됩니다:

```toml
[paths]
# 검색할 때 함께 포함되어야 할, wiki/ 폴더 바깥에 있는 개별 메모
# 파일들입니다. 예를 들어 최상위에 두는 인덱스나 변경 이력 파일 등을
# 여기에 적어두세요.
extra_root_pages = ["index.md", "log.md", "schema.md"]

[lint]
# 모든 메모의 맨 위에 있어야 할 정보 항목들입니다.
# 기본값은 ["type", "status", "created", "updated"]이고,
# "updated" 대신 "last_updated"를 쓰는 메모도 인정됩니다.
required_frontmatter = ["type", "status", "last_updated"]
```

이렇게 해도 기존 메모는 하나도 바꿀 필요가 없습니다.

## Obsidian과 함께 쓰기

모든 메모는 그냥 텍스트 파일이라서, `wiki/` 폴더를 인기 있는 노트 앱인
[Obsidian](https://obsidian.md/)에서 직접 열어볼 수도 있습니다. 그러면
Obsidian의 시각적 지도 보기, "여기로 연결된 링크" 보기, 검색 기능을
바로 그 메모들에 그대로 쓸 수 있습니다. AI가 이 메모들을 쓰는 방식은
전혀 포기할 필요가 없습니다.

메모끼리 링크가 연결되는 방식은 몇 가지 까다로운 예외까지 포함해서
Obsidian이 하는 방식과 맞도록 설계됐습니다:

- Obsidian의 모든 링크 스타일을 이해합니다. 다른 메모로의 평범한 링크,
  표시 이름을 따로 지정한 링크, 메모 안의 특정 제목으로 가는 링크,
  다른 메모의 내용을 그대로 가져오는 "임베드"까지 전부 포함됩니다.
- 폴더 경로가 포함된 링크는 Obsidian이 이해하는 방식과 똑같이
  이해합니다. 즉 링크가 적힌 메모 기준이 아니라 위키의 맨 위 폴더를
  기준으로 봅니다.
- 메모 맨 위에 "관련 메모"를 몇 가지 다른 형식으로 나열할 수 있는데,
  둘 다 올바르게 인식됩니다.
- 파일 이름에 공백이나 특수문자가 있는 링크(다른 도구에서 내보낸
  메모에서 흔함)도 정확히 이어집니다.
- `wiki/` 폴더 바깥을 가리키는 링크는 실제로 디스크에 그 파일이
  있는지 확인하기 때문에, 정말로 어디에도 존재하지 않을 때만 끊긴
  링크로 표시됩니다.

**실제 Obsidian과 약간 다른 점 몇 가지**: 이 도구는 위에서 말한
"관련 메모" 목록처럼, Obsidian의 지도 보기에는 나오지 않는 몇 가지
추가 연결 방식도 이해합니다. 이건 이 도구에만 있는 기능이기
때문입니다. 그리고 서로 다른 폴더에 있는 두 메모가 파일 이름이
완전히 같은 경우, 링크에서 어느 폴더의 파일을 말하는지 정확히
지정하지 않으면 이 도구도 Obsidian도 가끔 다른 쪽 파일을 잘못
고를 수 있습니다. 그 외에는 전부 똑같이 동작합니다.
